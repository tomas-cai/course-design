# 评审方法论详细参考

本文档提供 5 个评审维度的详细检查项、质量清单和深度分级标准。SKILL.md 定义流程骨架，本文档定义检查细节。

## 维度 1：结构完整性 —— 详细检查清单

### Phase 01 分析阶段检查
- [ ] 需求来源是否明确？
- [ ] 是否完成了痛点验证（不只有一个需求方确认）？
- [ ] 是否识别了不适合培训解决的问题？
- [ ] 学习者画像是否包含了情绪起点分析？
- [ ] 能力差距诊断是否基于布卢姆分层？

### Phase 02 设计阶段检查
- [ ] 每条学习目标是否使用"行为动词 + 可观测成果"格式？
- [ ] 目标层级是否匹配业务期望？
- [ ] 是否设计了情绪曲线？不是只有内容大纲？
- [ ] 每条目标是否有对应的内容和测评？
- [ ] 是否有至少一个"惊叫点"设计？
- [ ] 是否有认知负荷管理策略？

### Phase 03 开发阶段检查
- [ ] 五侧材料是否全部到位？（大纲/学员/讲师/实操/测评 + 课件PPT）
- [ ] 目标-内容-测评是否对齐？
- [ ] 实操案例是否有明确决策点？
- [ ] 测评形式是否匹配布卢姆层级？
- [ ] 课件（05-课程课件）是否已完成？
- [ ] 课件内容是否与课程设计方案和学习手册完全对齐？
- [ ] 每张课件是否一页一个要点、无大段文字？

### Phase 04 实施阶段检查
- [ ] 是否做了前置测评？
- [ ] 是否需要分层教学？
- [ ] 是否有随堂形成性评估计划？
- [ ] 讲师引导手册是否包含时间分配和关键话术？

### Phase 05 评估阶段检查
- [ ] 是否覆盖了柯氏 L1-L4？
- [ ] 是否对每项目标做了达成度分析？
- [ ] 是否复验了最初的"痛点假设"？
- [ ] 是否输出了优先级明确的迭代计划？

## 维度 2：内部一致性 —— 追踪路径

### 目标人群一致性
从 Phase 01 出发，依次检查：
1. Phase 01：学员画像中的"岗位与经验层级"
2. Phase 02：学习目标的难度层级是否匹配该人群
3. Phase 03：案例中的角色设定和场景复杂度是否匹配
4. Phase 04：前置测评的难度是否匹配

### 痛点一致性
从 Phase 00/01 出发：
1. Phase 00/01："核心痛点"的具体描述
2. Phase 02：情绪曲线和惊叫点设计是否回应了痛点
3. Phase 03：讲师话术中是否有针对性的痛点回应
4. Phase 05：痛点复盘机制是否覆盖了最初定义的痛点

### 学习目标一致性
从 Phase 02 出发：
1. Phase 02：目标清单中的每条 OBJ
2. Phase 03：每条 OBJ 是否有对应的测评（ASSESS）
3. Phase 05：每个 OBJ 是否有达成度分析

## 维度 3：内容深度 —— 分级定义

| 深度层级 | 定义 | 判断标志 |
|:---|:---|:---|
| **L1 骨架级** | 文件存在，有结构，有占位符 | 有"待填写"标记，表格有空行 |
| **L2 填充级** | 内容完整，可读性强 | 无占位符，每段落有实质性内容 |
| **L3 教学级** | 拿到就能上课，讲师不需额外准备 | 模拟一节课能讲下来，无盲区 |
| **L4 精品级** | 有视觉设计、录屏、课后扩展 | 学员体验完整，超出预期 |

**底线要求**：
- 全部产出物：L2 填充级
- 每张课件：L2 填充级
- Day 1 前 3 个模块：L3 教学级（讲师最难把握的部分）

## 维度 4：可教性 —— 检查细节

### 时间可行性
- [ ] 课程大纲中的时间段加起来 = 课程总时长
- [ ] 每个模块的"讲授时间"与"实操时间"比例合理
- [ ] 休息时间已预留
- [ ] 留白/问答时间已预留

### 讲师引导
- [ ] 讲师手册每个模块有时间分配
- [ ] 关键环节有逐字话术
- [ ] 常见学员反应有应对策略
- [ ] 技术环节有应急方案

### 案例可操作性
- [ ] 案例场景有充分上下文
- [ ] 决策节点有明确选项
- [ ] 参考答案有层次差异（好/中/差）
- [ ] 延展讨论有引导方向

### 测评可评分性
- [ ] 每条测评有评分标准
- [ ] 评分标准能区分优/中/差
- [ ] 权重分配合理（高层级测评 > 低层级测评）
- [ ] 有明确的"通过/不通过"分界线

## 维度 5：风险评估

### 风险类型清单

| 风险类型 | 例子 | 严重程度 |
|:---|:---|:---:|
| **技术依赖** | AI 工具不可用、API 限流、网络不稳定 | 高 |
| **讲师依赖** | 模块依赖某位讲师的经验才能讲好 | 中 |
| **学员差异** | 学员基础跨度大，分层后某一层不足 | 中 |
| **内容时效** | 引用工具/数据已过时 | 中 |
| **复杂度超限** | 模块内容太多，时间不够 | 低-中 |

### 风险评估模板

```
风险：[描述具体风险]
影响：[什么情况下哪些环节会出问题]
概率：高 / 中 / 低
严重程度：高 / 中 / 低
应对预案：[具体行动，不少于 2 条]
```

## 维度 3：内容深度 —— 新增检查项

### 实操案例质量检查

每个实操案例应包含以下要素：

| 要素 | 检查标准 | 常见缺失 |
|:---|:---|:---|
| **终端形态标注** | 明确标注 Mobile APP / PC Web / 纯评估练习 | 缺失 → 学员不知道按什么屏幕尺寸设计 |
| **推荐技术栈** | 给出 1-2 个推荐框架 + 选择理由 | 缺失 → 学员不知道该在 AGENTS.md 中声明什么 |
| **AGENTS.md 配置示例**（如适用） | 给出该技术栈的 AGENTS.md 声明片段 | 缺失 → 学员不知道规范怎么写 |
| **技术依赖** | 明确标注需要/不需要外部 API 或降级方案 | 缺失 → 开课时才发现工具不可用 |
| **教学价值与课程目标的对应** | 每个案例标明它对应课程的哪个核心教学维度 | 缺失 → 学员不知道为什么做这个案例 |

### 技术栈选型合理性检查

| 检查项 | 标准 |
|:---|:---|
| Tailwind CSS → C 端 PC Web | 简历工具、内容型前台页面 ✅ |
| Nuxt UI / Tailwind 移动端优先 → Mobile APP | 面试训练、碎片场景工具 ✅ |
| Ant Design Pro / Element Plus → 后台管理 | CRM、数据后台、运营后台 ✅ |
| 技术栈是否与 AGENTS.md 中的 UI 约束一致？ | AGENTS.md 中声明的框架必须与推荐一致 |

## 维度 4：可教性 —— 新增检查项

### 技术依赖预警

- [ ] 每个构建型案例是否标注了"推荐技术栈"？
- [ ] 推荐的框架是否需要学员课前预习？（如 Ant Design 概念）
- [ ] 如果需要预习，是否有课前资料准备清单？
- [ ] 是否有多框架备选方案？（如学员不熟悉 React 可以用 Vue 版 Element Plus）
- [ ] 是否标注了"技术依赖"（外部 API / AI 工具 / 降级方案）？
