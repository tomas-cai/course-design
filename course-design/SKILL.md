# 磨课 (Course Design) Skill

## 核心理念

**把课程当作产品做**。学习者就是用户，教学就是体验设计，而不仅是知识传递。

## 项目结构与产出物流

Skill 的核心设计：**每个阶段产出唯一的标准化文件，下一阶段以此为输入**。这类似 superpowers——每个技能独立可调用，产出物是阶段间的契约。

```
<course-project-root>/             ← 课程项目根目录（用户指定）
│
├── 00-培训需求spec.md            ← 来自 course-discovery → Phase 01 读取
├── 01-需求定位报告.md            ← Phase 1 产出 → Phase 2 读取
├── 02-课程设计方案.md            ← Phase 2 产出 → Phase 3 读取
├── 03-教学材料包/                ← Phase 3 产出（目录）→ Phase 4 读取
│   ├── 00-课程大纲.md
│   ├── 01-学员手册.md
│   ├── 02-讲师手册.md
│   ├── 03-实操案例.md
│   └── 04-测评方案.md
│   └── 05-课程课件               ← 讲师授课用 PPT / 可视化材料

├── 04-实施记录与反馈.md          ← Phase 4 产出 → Phase 5 读取
└── 05-评估报告与迭代计划.md      ← Phase 5 产出（完结）
```

### 产出物契约

每个产出物是自包含文件（或目录），含两部分：
1. **元数据头**（结构化摘要：课程名、阶段、完成日期）
2. **内容正文**（本阶段完整产出）

产出物的具体格式由各阶段参考文件中的"产出物结构"章节定义。

## 入口与阶段选择

当用户提出课程开发需求时：

1. **已有课程项目** → 询问项目目录路径，检出已有产出物判断进度
2. **新课程项目** → 询问存放路径 + 课程基本信息，从分析阶段开始
3. **指定中间阶段**（如"我已有需求分析报告"）→ 读取用户提供的文件 → 从对应阶段开始

### 进度判定规则

```
检出 00-培训需求spec.md → 不存在 → 告知用户：可用 course-discovery Skill 做需求探索，或直接开始 Phase 01
检出 00-培训需求spec.md → 存在 → 自动读取作为 Phase 01 输入

检出 01-需求定位报告.md → 不存在(且无00) → 从 Phase 01 开始
检出 01-需求定位报告.md → 存在   → 问用户是否继续 Phase 02
检出 02-课程设计方案.md → 存在   → 问用户是否继续 Phase 03
检出 03-教学材料包/    → 存在   → 问用户是否继续 Phase 04
检出 04-实施记录与反馈.md → 存在  → 问用户是否继续 Phase 05
检出 05-评估报告与迭代计划.md → 存在 → 课程已完成，问是否开新课或迭代
```

**原则**：不让用户一开始就定义所有细节——设计的价值在分析中逐步收敛。也不强迫用户从头开始——已有产出物应该复用。

## 阶段执行节奏

每个阶段有两种执行模式，进入阶段时自动判断。判断依据：进入阶段时用户的消息风格。

| 用户特征 | 使用模式 | 做法 |
|:---|:---|:---|
| 消息中包含多个信息点，能一次性回答多个问题 | **批量模式**（默认） | 将参考文件中的"核心问题"一次性抛出，用户集中回答后统一处理 |
| 消息简短、表示不确定、或被问到才知道 | **逐题模式** | 按参考文件中的"探索问题"逐个深入追问 |

各阶段的"核心问题"和"探索问题"定义在该阶段 reference 文件头部的"节奏选择"章节。

## 理论基础（概述）

六维嵌套模型。详见 `references/overview.md`，首次使用时读取。

| 层次 | 维度 | 解决的问题 |
|:---|:---|:---|
| **流程层** | ADDIE | 项目管控 |
| **质量层** | 迪克-凯瑞 | 内容对齐 |
| **精度层** | 布卢姆 | 目标精度 |
| **体验层** | LXD + 产品思维 | 学习动机与完课率 |
| **敏捷层** | SAM/AGILE | 内容时效性 |
| **评估层** | 柯氏四级 | 效果闭环 |

**痛点**：学习者的真实困难与焦虑 / **惊叫点**："原来如此！"的顿悟时刻 / **爽点**：顺畅有掌控感的细节体验

## 阶段工作流

每个阶段三块描述：**前置输入 → 执行内容 → 产出物**。完成后产出物写盘，下次对话或另一实例可读取继续。

### Phase 01：分析阶段

**前置输入**：`00-培训需求spec.md`（如果已通过 course-discovery 产出）或无（新项目直接开始）

**执行**：读 `references/phase-01-analysis.md`，引导用户：
1. 业务需求拆解（痛点验证）
2. 学习者画像与情绪分析
3. 能力差距诊断（布卢姆分层）

**核心问题**：学习者的痛点是什么？他们为什么愿意花时间学这门课？

**产出物**：写入 `01-需求定位报告.md`（格式见 phase-01-reference 的"产出物结构"）

**阶段完成条件**：
- ✅ `01-需求定位报告.md` 已写入课程项目目录
- ✅ 已执行或已有明确计划的 P1 验证（3 名目标学员访谈，验证需求假设）
- ✅ 分析阶段检查清单（见 `references/common-tools.md`）全部通过



### Phase 02：设计阶段

**前置输入**：读 `01-需求定位报告.md`（无则退回 Phase 01）

**执行**：读 `references/phase-02-design.md`，引导用户：
1. 编写学习目标（布卢姆动词 + 可观测成果）
2. 设计学习体验旅程（情绪曲线）
3. 目标-内容-评估对齐矩阵
4. 叙事弧与认知节奏
5. 痛点/惊叫点/爽点检查

**核心问题**：惊叫点在哪里？学习旅程中有没有认知高潮？

**产出物**：写入 `02-课程设计方案.md`

**阶段完成条件**：
- ✅ `02-课程设计方案.md` 已写入课程项目目录
- ✅ 已执行或已有明确计划的 P2 验证（1 小时最小原型课试讲给 2-3 人）
- ✅ 设计阶段检查清单（见 `references/common-tools.md`）全部通过


### Phase 03：开发阶段

**前置输入**：读 `02-课程设计方案.md`（无则退回 Phase 02）

**执行**：读 `references/phase-03-development.md`，引导用户：
1. 教学内容开发（四侧材料）
2. 实操案例设计
3. 测评方案设计
4. 视觉与交互检查
5. 开发中期 P3 敏捷验证

**产出物**：创建 `03-教学材料包/` 目录，其下产出 6 个文件（含课件PPT）

**阶段完成条件**：
- ✅ `03-教学材料包/` 目录已创建，5 个文件齐全
- ✅ 已执行或已有明确计划的 P3 验证（前 20% 内容试课给 3-5 人）
- ✅ 开发阶段检查清单（见 `references/common-tools.md`）全部通过


### Phase 04：实施阶段

**前置输入**：读 `03-教学材料包/` 中的材料；读 `02-课程设计方案.md` 重温设计决策

**执行**：读 `references/phase-04-implementation.md`，引导用户：
1. 前置测评与分层
2. 随堂验证与反馈收集
3. 讲师引导手册设计

**核心问题**：学员的"爽点"是什么？节奏是否舒适？

**产出物**：写入 `04-实施记录与反馈.md`

**阶段完成条件**：
- ✅ `04-实施记录与反馈.md` 已写入课程项目目录
- ✅ 实施阶段检查清单（见 `references/common-tools.md`）全部通过


### Phase 05：评估阶段

**前置输入**：读 `04-实施记录与反馈.md`；读 `01-需求定位报告.md` 重温痛点假设

**执行**：读 `references/phase-05-evaluation.md`，引导用户：
1. 目标达成度分析
2. 柯氏四级评估（L1-L4）
3. 痛点假设复盘
4. 课程迭代优化清单

**产出物**：写入 `05-评估报告与迭代计划.md`。至此一轮课程开发完成。

**阶段完成条件**：
- ✅ `05-评估报告与迭代计划.md` 已写入课程项目目录
- ✅ 评估阶段检查清单（见 `references/common-tools.md`）全部通过
- ✅ 本轮课程开发全流程完成，产出物可被下一轮课程参考复用


## PM + AI 课程的特殊处理

当课程涉及 PM 和 AI/前端时，读 `references/pm-ai-adaption.md`。

关键差异：
- 判断类能力用"案例判断 + 同行评议 + 项目产出"替代考试
- 目标集中在布卢姆高层级（分析→评价→创造）
- 内容按稳定性分层：底层原理（年更）/ 工具方法论（季更）/ 实操示例（月更）


## 跨 Skill 协作

本 Skill 与以下 Skill 通过产出物文件协作：

| Skill | 输入 | 产出 | 协作方式 |
|:---|:---|:---|:---|
| **course-discovery** | 行业观察 / 市场信号 | `00-培训需求spec.md` | 课程方向探索 → 本 Skill 读取后开始 Phase 01 |
| **course-review** | `01~05` 产出物 | 评审报告 | 本 Skill 交付课程后，用 course-review 做质量评审 |

**协作机制**：文件命名约定（`00-*`, `01-*`, ...）是跨 Skill 的接口契约。任意 Skill 产生的输出文件均可被其他 Skill 消费，无需互相导入。

## 与 superpowers 的关系

| superpowers 概念 | 本 Skill 的对应 |
|:---|:---|
| **独立技能** | 每个阶段可单独调用 |
| **产出物契约** | 01→02→03→04→05 文件依赖链 |
| **入口接续** | 检出已有产出物后从对应阶段切入 |
| **跨项目复用** | 评估报告可被另一课程参考 |

**典型场景**：从头开发完整走 5 阶段 / 已有设计方案从开发开始 / 读另一课评估报告做参考 / 单做一轮需求分析产出 `01-需求定位报告.md` 即结束。

### 产出物通用规范

所有产出物文件使用 Markdown：
1. H1 标题：`# [产出物名称]：[课程名称]`
2. 用表格、分割线、三级标题结构化组织
3. 关键决策点用 `**` 加粗标记
4. 待办事项用 `> **待办**` 引用块
5. 末尾用 `---` 分割线 + `## 本阶段完成检查` 检查清单
