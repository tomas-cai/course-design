# 开发阶段参考（Phase 03 - Development）

## 节奏选择

进入本阶段时，先判断用户状态来决定执行节奏：

| 用户特征 | 使用模式 | 做法 |
|:---|:---|:---|
| 消息中包含多个信息点，能一次性回答多个问题 | **批量模式**（默认） | 将下面的核心问题一次性抛出，用户集中回答后统一处理 |
| 消息简短、表示不确定、需要引导 | **逐题模式** | 按下面的探索问题逐个深入追问 |

### 批量模式：核心问题

- 每个模块的核心内容方向
- 实操案例的场景选择（几个案例？难度递进？）
- 测评形式与评分标准的细节确认

### 逐题模式：深入探索

- 讲师话术的关键节点与逐字稿
- 视觉设计的风格要求
- 敏捷验证节点（P2/P3）的排期
- 案例的参考答案不同水平分级

## 第一节：教学内容开发规范

### 四侧材料体系

每门课程需要开发四类材料：

| 材料侧 | 内容 | 受众 | 关键标准 |
|:---|:---|:---|:---|
| **学员侧** | 讲义、课件、学习手册 | 学习者 | 自学友好、结构清晰、重点标注 |
| **讲师侧** | 讲师引导手册、逐字稿 | 讲师 | 含时间把控、关键话术、应急预案 |
| **实操侧** | 案例说明、练习指南、参考答案 | 学习者+讲师 | 场景真实、步骤明确、难度递进 |
| **测评侧** | 测评题目、评分标准、答案解析 | 学习者+讲师 | 目标-测评对齐、评分标准可操作 |

### 内容写作原则

1. **每个模块一个核心观点**：不要在一个模块里塞多个独立概念
2. **先建立"为什么"再讲"是什么"**：先说明这个知识点解决了什么痛点
3. **使用案例贯穿**：每个抽象概念配一个具体案例
4. **图表优先于文字**：能用图/表/模型表达的就不要用段落
5. **总结+预告**：每个模块结束时做要点回顾并预告下个模块

### 跨模块技能的设计原则

有些技能不属于任何一个单独的模块，而是需要贯穿多个模块（如 Git 版本管理、
文档治理、AI 工具操作习惯）。处理原则：

| 错误做法 | 正确做法 |
|:---|:---|
| 单独开一堂"Git 课" | **分散融入**：首次用到时教基础操作，后续每次用到时加深 |
| 只在某一天讲一次 | **螺旋上升**：M1 知道概念 → M3 首次实践 → M5 系统讲解 → M7 实战应用 |
| 不教就要求用 | **搭脚手架**：模板化操作步骤，学员先复制再理解 |

**以 Git 为例的正确融入方式**：
- M3（首次接触代码）：教 git init / add / commit 三个命令就够了
- M4-3（产出规范文件后）：首次 commit，体验版本记录的价值
- M5（专题讲解）：git push / status / log，完整工作流
- M6（全链路项目）：全程用 Git 管理，验收时看 commit 历史

**判断标准**：学员在课程结束时，对这些技能的掌握程度应该达到"独立使用"的水平，
而不是"听说过"。


### 内容质量检查

- 每条内容是否有明确对应的学习目标？
- 每条内容是否有明确对应的测评方式？
- 内容是否经过"痛点-惊叫点-爽点"检查？
- 文本量是否匹配预期时长？（通常每分钟 150-200 字口语讲解对应一页课件内容）

## 第二节：实操案例设计模板

### 案例设计原则

1. **场景驱动而非工具驱动**：不是"这个工具有 ABC 功能"，而是"在场景 D 中你应该如何用这个工具"
2. **有决策点**：好的案例在节点上需要学习者做判断，不只是机械执行
3. **真实但脱敏**：基于真实业务场景但做脱敏处理
4. **难度递进**：从"引导式案例"→"半开放式案例"→"完全开放式案例"
> **案例多样性检查**：多个案例之间必须使用不同的需求场景——不要用同一个场景出两个案例。
开场案例、模块练习案例、全链路项目案例应覆盖不同的产品类型
（如 C 端工具 / B 端后台 / 移动端等），确保学员在不同场景下都有练习机会。


### 案例结构

| 要素 | 设计说明 |
|:---|:---|
| **场景描述** | 发生了什么？提供充分的上下文信息 |
| **决策节点** | 需要学习者判断什么？有哪几个可能选项？ |
| **成功标准** | 什么算做对了？什么算做得不够好？ |
| **参考答案** | 给出 2-3 种不同水平的解答，便于讲师做差异化点评 |
| **延展讨论** | 如果场景参数变了，结论是否改变？ |

## 第三节：测评方案设计

### 布卢姆层级与测评形式匹配

| 层级 | 推荐测评形式 | 示例 |
|:---|:---|:---|
| **记忆** | 选择题、判断题、填空题 | "以下哪个是 AI 的常见应用？" |
| **理解** | 简答题、概念图、类比说明 | "用你自己的话解释大语言模型的工作原理" |
| **应用** | 实操题、模拟场景题 | "用 Copilot 为这个需求生成一个高保真原型" |
| **分析** | 案例比较、归因分析 | "分析这两个方案各有什么优劣？为什么？" |
| **评价** | 方案评估、同行评议 | "评估这三份竞品分析报告，哪一份最有效？给出评分和理由" |
| **创造** | 项目产出、设计方案 | "为这个产品需求设计一个完整的 AI 功能方案" |

### PM 课程测评特殊规则
- 判断类能力使用"案例判断 + 同行评议 + 项目产出"替代传统考试
- 每个测评必须附带评分标准，不能只问开放题不评
- 详见 `references/pm-ai-adaption.md`

## 第四节：视觉与交互体验检查

### 课件视觉设计原则

1. **降低认知负荷**
   - 一页只讲一个要点
   - 使用留白和视觉层次（标题>副标题>正文>注释）
   - 避免大段文字（超过 30 字就考虑换图或清单）

2. **信息可视化**
   - 逻辑关系用图表（流程→流程图，对比→表格，因果关系→鱼骨图）
   - 关键数据用可视化（数字不要藏在段落里）

3. **交互设计**（如果是线上课程）
   - 明确的进度提示（当前在哪个模块、还有多久）
   - 可点击的索引/目录
   - 每个交互步骤有即时反馈

### 课程课件内容规范

课件（PPT/可视化材料）是 Phase 03 的第 6 个产出文件 `05-课程课件`。

**内容来源规则**：课件内容的唯一来源是 `02-课程设计方案.md` 和 `01-学员手册.md`。
不允许添加这两份文件之外的新内容——如果讲师发现"有个内容想加但手册没写"，先回去改手册。

**整套课件必须包含的页面**：

| 类型 | 数量 | 内容来源 |
|:---|:---:|:---|
| 课程封面 | 1 | 课程名、讲师、日期 |
| 课程全景 | 1 | 从 `02-课程设计方案.md` 的情绪曲线提取 |
| 模块标题页 | 每模块 1 页 | 模块名、时长、学习目标 |
| 核心内容页 | 每模块 2-5 页 | 从 `01-学员手册.md` 提取关键概念、模型框架、对比表 |
| 实操指引页 | 每个练习 1 页 | 练习说明、成功标准、时间限制 |
| 惊叫点触页 | 每个惊叫点 1 页 | 从 `02-讲师手册.md` 提取关键话术和触发词 |
| 总结页 | 每半天 1 页、课程末尾 1 页 | 关键要点回顾 |

**质量控制**（对齐检查）：
- 每张课件的 H2/H3 标题 → 能在 `02-课程设计方案.md` 中找到对应模块
- 每张课件的核心观点 → 能在 `01-学员手册.md` 中找到对应段落
- 每个练习页面 → 能在 `03-实操案例.md` 中找到对应案例
- 完成整份课件后 → 对照 `02-课程设计方案.md` 的"学习目标清单"逐条确认每条目标都有对应的课件页

**视觉质量**（见上方"课件视觉设计原则"）：
- 一页一个要点
- 避免大段文字（超过 30 字换图或清单）
- 关键关系用图表（流程→流程图，对比→表格，因果→图）
- 对齐课程节奏——每 15-20 分钟应有 1 张"切换"页（休息/讨论/实操标识）

## 第五节：敏捷原型验证

### 三个验证节点

| 节点 | 时机 | 验证什么 | 方式 | 耗时 |
|:---|:---|:---|:---|:---|
| **P1** | 分析→设计之间 | 需求假设是否成立？ | 3 个目标学员的 30 分钟访谈 | 2 天 |
| **P2** | 设计→开发之间 | 体验旅程是否有效？结构合理？ | 1 小时最小原型课（PPT+1 个关键练习）+ 2-3 人试听 | 3 天 |
| **P3** | 开发中期（~50% 完成） | 前 20% 内容是否达到预期效果？ | 完整试课 3-5 名目标学员 | 5 天 |

### P3 验证的具体操作

完成约 50% 的内容开发后，执行 P3 验证：
1. 选定课程前 20% 的内容（通常是引言 + 第一个核心模块）
2. 邀请 3-5 名目标学员做一次完整的模拟授课
3. 收集反馈重点：
   - 节奏是否合适？（太快 / 太慢 / 刚好）
   - 认知负荷是否合理？（太多信息 / 太少 / 刚好）
   - 第一个"惊叫点"是否真的产生了效果？
   - 学习者是否理解了课程价值？
4. 根据反馈调整剩余 50% 的内容设计

### 03-教学材料包/01-学员手册.md

内容结构：每个模块一章，含核心概念、案例说明、关键要点、课后练习。
> **学员手册内容标准**：手册不仅要列出知识点纲要，还需包含：
(1) **完整工作示例**——每条方法论配 2-3 个逐词对比（好 Prompt vs 差 Prompt），
学员拿回去能直接复制使用。(2) **分步骤实操指引**——每个练习的逐步骤指南，
含"预期输出"截图或描述。(3) **自学扩展**——课后可独立完成的拓展练习和参考阅读。
学员手册的目标是：学员课后拿到手册能独立复现课程中的重点练习，无需依赖讲师。
