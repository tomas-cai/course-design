# 通用工具参考（Common Tools）

贯穿全流程的工具和方法。Codex 在对应步骤中按需读取。

## 布卢姆行为动词手册

### 完整动词表

| 层级 | 常用动词 | PM 课程推荐动词 | 不推荐（太模糊） |
|:---|:---|:---|:---|
| **记忆** | 列出、识别、命名、复述、定义 | 列举、识别 | 了解、熟悉 |
| **理解** | 解释、总结、举例、分类、推断 | 解释、总结、类比 | 理解、掌握 |
| **应用** | 执行、实施、使用、演示、操作 | 使用、制作、演示 | 运用 |
| **分析** | 区分、组织、归因、比较、拆解 | 比较、分析、拆解 | 分析清楚 |
| **评价** | 评估、判断、辩护、推荐、排序 | 评估、判断、推荐 | 评价 |
| **创造** | 设计、构建、规划、开发、制定 | 设计、规划、提出方案 | 创新 |

### 动词选择原则

1. 动词必须**可观测**——别人能看到学员完成了这个动作
2. 动词必须**可测评**——有人能用评分标准评价完成质量
3. 优先用**具体动词**避免**抽象动词**
   - ❌ "学员能理解用户需求"（理解不可直接观测）
   - ✅ "学员能**列出**至少 5 种用户需求分析方法，并**选择**一种**设计**一份访谈提纲"（三个具体行为可观测）

## 敏捷验证节点模板

### P1 验证（分析→设计之间）

验证问题：
- 学员是否真的有这个痛点？痛到什么程度？
- 学员对课程主题的预期是什么？
- 我们设计的解决方案方向是否对？

操作：
- 找 3 个目标学员做 30 分钟访谈
- 不需要任何教学内容，只需要一个课程概述（1 页纸）

### P2 验证（设计→开发之间）

验证问题：
- 课程结构是否合理？模块之间逻辑是否自然？
- 体验旅程是否有效？情绪曲线是否合理？
- 第一个"惊叫点"设计是否有效？

操作：
- 准备 1 小时的最小原型课程（PPT + 一个关键练习）
- 找 2-3 人做一次试讲
- 收集反馈后调整设计

### P3 验证（开发中期~50%）

验证问题：
- 前 20% 的内容是否达到了预期学习效果？
- 节奏和认知负荷是否合理？
- 是否有需要大规模调整的地方？

操作：
- 用已开发的前 20% 内容做一次完整试课
- 3-5 名目标学员
- 收集反馈，决定剩余 50% 是否需要调整方向

## 课程开发质量检查总清单

### 分析阶段检查
- [ ] 需求来源是否明确？
- [ ] 是否完成了痛点验证（不只有一个需求方确认）？
- [ ] 是否识别了不适合培训解决的问题？
- [ ] 学习者画像是否包含了情绪起点分析？
- [ ] 能力差距诊断是否基于布卢姆分层？

### 设计阶段检查
- [ ] 每条学习目标是否使用"行为动词 + 可观测成果"格式？
- [ ] 目标层级是否匹配业务期望？
- [ ] 是否设计了情绪曲线？不是只有内容大纲？
- [ ] 每条目标是否有对应的内容和测评？
- [ ] 是否有至少一个"惊叫点"设计？
- [ ] 是否有认知负荷管理策略？

### 开发阶段检查
- [ ] 四侧材料是否全部到位？（学员/讲师/实操/测评）
- [ ] 目标-内容-测评是否对齐？
- [ ] 实操案例是否有明确决策点？
- [ ] 测评形式是否匹配布卢姆层级？
- [ ] 课件（05-课程课件）是否已完成？
- [ ] 课件内容是否与课程设计方案和学习手册完全对齐？
- [ ] 每张课件是否一页一个要点、无大段文字？
- [ ] 是否执行了至少一次敏捷验证？（P1/P2/P3）？

### 实施阶段检查
- [ ] 是否做了前置测评？
- [ ] 是否需要分层教学？
- [ ] 是否有随堂形成性评估计划？
- [ ] 讲师引导手册是否包含时间分配和关键话术？

### 评估阶段检查
- [ ] 是否覆盖了柯氏 L1-L4？
- [ ] 是否对每项目标做了达成度分析？
- [ ] 是否复验了最初的"痛点假设"？
- [ ] 是否输出了优先级明确的迭代计划？

### L1 反应层
- [ ] 收集整体满意度评分
- [ ] 收集 NPS（净推荐值）
- [ ] 收集痛点解决度反馈
- [ ] 收集惊叫点命中率反馈
- [ ] 收集爽点体验反馈

### L2 学习层

- [ ] 对比前置测评数据

### L3 行为层

- [ ] 计算培训 ROI


> 以下五个维度构成比检查清单更全面的评审模型。建议在每个课程项目交付后按顺序执行。

### 1. 结构完整性检查

验证产出物流是否完整，每个阶段的产出物是否存在：

| 检查 | 标准 |
|:---|:---|
| 00-* | 如执行过 Phase 00，`00-*.md` 应存在，含信号摘要和课程假设 |
| 01-* | `01-*.md` 应存在，含需求来源、痛点验证、学习者画像、能力差距 |
| 02-* | `02-*.md` 应存在，含学习目标、情绪曲线、对齐矩阵 |
| 03-*/ | `03-*/` 应存在，含课程大纲、学员手册、讲师手册、实操案例、测评方案 |
| 04-* | `04-*.md` 应存在，含前置测评、随堂验证设计 |
| 05-* | `05-*.md` 应存在，含柯氏评估、迭代计划 |
| 每个产出物的"本阶段完成检查" | 检查项应为已勾选状态（[x]），未勾选的需说明原因 |

### 2. 内部一致性检查

验证跨阶段的信息是否一致，这是最容易出现断裂的地方：

| 检查项 | 对应关系 | 典型断裂 |
|:---|:---|:---|
| 目标人群 | Phase 01 画像 → Phase 02 目标 → Phase 03 案例 → Phase 04 前置测评 | 画像写的是"零经验毕业生"，但案例假设了 2 年工作经验 |
| 学习目标 | Phase 02 目标 → Phase 03 测评 → Phase 05 达成度 | OBJ 在 Phase 02 定义了，但在 Phase 03 没有对应的测评 |
| 情绪基线 | Phase 01 情绪 → Phase 02 情绪曲线 → Phase 03 讲师话术 | Phase 01 说"焦虑"，但讲师手册没有针对焦虑设计话术 |
| 痛点假设 | Phase 00/01 痛点 → Phase 05 痛点复盘 | Phase 05 复盘的问题在 Phase 01 没有定义过 |

**检查方法**：挑三个关键字段（目标人群 / 核心痛点 / 课程目标），从 Phase 01 开始沿着产出物流往下看，每到一个阶段停下来确认"这个阶段的文件是否还认同一个字段"。

### 3. 内容深度检查

比"文件存在"更进一步——文件的内容是否达到了"可教学"的质量：

| 深度层级 | 定义 | 检查方法 |
|:---|:---|:---|
| **L1** 骨架级 | 有文件，有结构，有占位符 | 文件能打开 |
| **L2** 填充级 | 内容完整，可读性强 | 逐段阅读，没有"待填写"标记 |
| **L3** 教学级 | 拿到就能上课，讲师不需要额外准备 | 模拟一节课——只读手册，能讲吗？ |
| **L4** 精品级 | 有视觉设计、有录屏、有课后扩展 | 学员体验完整，超出预期 |

**目标**：课程交付时至少达到 L2（填充级），关键模块（如第 1 天的第 1 个模块）应达到 L3（教学级）。

### 4. 可教性检查

模拟"明天就要上课"场景，验证材料是否可执行：

| 检查项 | 操作方式 |
|:---|:---|
| 1. 时间表是否可执行 | 把 00-课程大纲.md 的每段时间加起来，看是否等于总时长 |
| 2. 讲师能否照着讲 | 打开讲师手册，从 M1 到结束，逐段读一遍——"这里有盲区吗？" |
| 3. 案例是否可操作 | 还原案例场景，确认决策点有明确选项，参考答案有差异层次 |
| 4. 测评是否可评分 | 拿学员的 3 份不同水平作业试评，看评分标准能否区分出优/中/差 |
| 5. 技术依赖是否确认 | 实操环节用到的工具/平台/API 是否都已确认可运行 |

### 5. 风险评估

识别课程交付前最可能出问题的 3 件事：

```
风险 1：[如"M4-1 自由手搓依赖 AI 工具可用性"]
  影响：如果 Cursor 不可用，整个 M4 叙事弧断裂
  概率：低
  应对：准备本地演示录像作为降级方案

风险 2：[..."...."]
  影响：...
  概率：...
  应对：...
```

**风险来源**：讲师经验（没讲过的环节）、技术依赖（AI 工具、API）、学员差异（跨度大的分层）、内容时效性（过时的案例）。
