# 设计阶段参考（Phase 02 - Design）

## 节奏选择

进入本阶段时，先判断用户状态来决定执行节奏：

| 用户特征 | 使用模式 | 做法 |
|:---|:---|:---|
| 消息中包含多个信息点，能一次性回答多个问题 | **批量模式**（默认） | 将下面的核心问题一次性抛出，用户集中回答后统一处理 |
| 消息简短、表示不确定、需要引导 | **逐题模式** | 按下面的探索问题逐个深入追问 |

### 批量模式：核心问题

- 学习目标清单（每条含布卢姆层级的动词 + 可观测成果）
- 模块数量与时长的初步分配
- 测评方式偏好（传统考试/案例判断/项目产出/同行评议）

### 逐题模式：深入探索

- 每个模块的具体活动设计
- 情绪曲线设计（什么模块是高负荷、哪里是休息点）
- 叙事弧的起承转合结构
- 痛点/惊叫点/爽点的逐模块检查

## 第一节：学习目标编写

### 目标编写公式

一个标准的学习目标 = **行为动词 + 内容对象 + 可观测成果**。

**格式**："学员能够[布卢姆层级动词] [内容对象]，[可观测成果]"

**示例**：
- ❌ "学员了解AI工具"（不可观测）
- ✅ "学员能够**区分**AI原型生成工具和AI代码补全工具的**适用场景**，**给出**至少3个选择依据"（分析层，可观测）
- ✅ "学员能够**设计**一个基于AI的前端产品功能，**输出**包含需求说明和原型链接的产品提案"（创造层，可观测）

### 布卢姆层级选择指南

| 期望行为 | 推荐层级 | 典型动词 |
|:---|:---|:---|
| 知道概念/术语 | 记忆 | 列出、识别、命名、复述 |
| 理解原理 | 理解 | 解释、总结、举例、分类 |
| 能用工具操作 | 应用 | 执行、实施、使用、演示 |
| 能判断方案优劣 | 分析 | 区分、组织、归因、比较 |
| 能选择最佳方案 | 评价 | 评估、判断、辩护、推荐 |
| 能创造新方案 | 创造 | 设计、构建、规划、开发 |

### PM 课程的特殊规则

PM 课程的目标往往涉及"判断力"这类非操作类能力。处理方式：
- **判断类能力**：不用"了解/理解/掌握"这类模糊动词，改用"分析/评估/比较/判断/推荐/设计"
- 与其说"学员能理解产品需求优先级"（不可观测），不如说"学员能**评估**三个需求场景的**优先级排序**，**给出**排序依据"
- 详见 `references/pm-ai-adaption.md`

## 第二节：学习体验旅程地图

### 基本思路

像用户体验地图一样设计学习旅程。不是课程大纲（大纲只列知识点），而是**学习者在每个时间切片上的情绪曲线、行为路径和接触点设计**。

### 模板结构

将课程按时间线拆为 3-8 个**核心模块**（不是课时，是体验节点），每个模块包含：

| 维度 | 设计内容 |
|:---|:---|
| **预计时长** | 分钟 |
| **认知层级** | 记忆/理解/应用/分析/评价/创造 |
| **学习目标** | 对应目标编号 |
| **核心活动** | 听讲 / 讨论 / 实操 / 案例分析 / 测评 / 反思 |
| **情绪曲线** | 该模块期望的情绪状态（兴奋/好奇/专注/挑战/成就感） |
| **认知负荷** | 高 / 中 / 低（控制节奏） |
| **关键体验设计** | 痛点解决 / 惊叫点设计 / 爽点营造 |

### 情绪曲线设计原则

- 不要全部高负荷（认知疲劳）也不要全部低负荷（无聊）
- 理想节奏：低（热身）→ 高（知识输入）→ 高（认知冲突/惊叫点）→ 中（应用练习）→ 低（反思回顾）
- 每个"高负荷"模块后安排一个"低负荷"模块作为缓冲
- 惊叫点通常在课程中后段，不在开头（需要先建立认知框架才能打破它）

## 第三节：目标-内容-评估对齐矩阵

这是迪克-凯瑞模型的核心校验工具。

### 对齐矩阵

| 学习目标编号 | 目标描述 | 布卢姆层级 | 对应教学内容 | 对应测评方式 | 对齐状态 |
|:---|:---|:---|:---|:---|:---|
| OBJ-01 | ... | 应用 | Module 2 | 实操考核 | ✅ |
| OBJ-02 | ... | 分析 | Module 3-4 | 案例分析 | ❌ 缺测评 |

### 检查规则

1. **无内容的目标**：每条目标是否都有对应的教学内容覆盖？→ 如果没有，目标太宽泛或内容缺失
2. **无测评的内容**：每条教学内容是否都有对应的测评方式？→ 如果没有，学没学到没人知道
3. **层级错配**：目标定在"应用"层，但测评只测"记忆"层？→ 测评设计需要升级

> 对 PM 的判断类课程，高层级测评可用"案例判断 + 同行评议 + 项目产出"替代传统考试题。

## 第四节：叙事弧与认知节奏设计

### 起承转合结构

| 阶段 | 对应的课程环节 | 体验目标 | 时长占比 |
|:---|:---|:---|:---|
| **起** | 开场 → 痛点唤醒 | 建立"为什么学"的紧迫感 | ~10% |
| **承** | 知识构建 → 概念讲解 | 建立认知框架 | ~30% |
| **转** | 认知冲突 → 惊叫点 | 打破旧认知，创造顿悟 | ~20% |
| **合** | 应用检验 → 回顾升华 | 成就感 + 闭环感 | ~40% |

### 认知节奏设计要点

1. **每个模块有明确"信息量"上限**：一次只讲 3-5 个新概念，超过就是认知过载
2. **每 15-20 分钟切换一次学习活动**：讲→练→讨论→讲→练
3. **留白时间**：每 45 分钟课程至少留 5 分钟给学习者消化和提问
4. **前置知识检查**：在讲新概念前先确认学习者的先决知识是否到位

## 第五节：痛点/惊叫点/爽点设计检查表

### 定义回顾

| 概念 | 定义 | 课程场景示例 |
|:---|:---|:---|
| **痛点** | 学习者在学习前/学习中遇到的真实困难、焦虑、挫败 | 担心学不会、觉得跟实际工作无关、基础断层跟不上 |
| **惊叫点** | 让学习者发出"原来如此！"的顿悟时刻 | 一个反转案例、一个颠覆常识的原理揭示、一次"我竟然也能做出来"的实操收获 |
| **爽点** | 让学习过程顺畅、有掌控感的细节体验 | 恰到好处的提示、即时的正向反馈、清晰的进度感知、美观易用的课件 |

### 逐模块检查

| 检查项 | 模块1 | 模块2 | 模块3 | ... |
|:---|:---|:---|:---|:---|
| **痛点解决**：本模块解决了学习者的什么具体痛点？ | | | | |
| **惊叫点设计**：学习者会在哪一个环节产生"原来如此"的感觉？ | | | | |
| **爽点营造**：本模块在体验细节上做了什么让学习顺畅的设计？ | | | | |
| **认知负荷合理**：本模块的学习强度是否适当？ | | | | |
